# yutai-expiry ユースケース棚卸し

## 目的

優待期限帳の改善検討時に、想定漏れによる設計の甘さを避けるためのユースケース正本。
新機能・データモデル変更・UI 改修を検討するときは、このリストでカバー範囲をチェックする。

関連: [yutai-expiry UI / data model backlog](./yutai-expiry-ui-data-model.md)

## 設計に効く前提

- 株主優待は半期 / 通期ごとに **同銘柄から繰り返し** 届く性質を持つ
- 個別アイテム単位の操作だけでなく、「取得 → 使う → 取得」の **継続サイクル** が主要文脈
- データは端末内 (localStorage) のみで完結する（前提固定）
- 主たる利用端末はモバイル (Android Chrome) + PC (Windows Chrome)

## 現状設計の構造的な穴

継続サイクル視点で見たときに弱い領域:

1. **「銘柄」概念が無い**
   個別アイテムの集合でしか表現できないため、「同銘柄を継続している」が一覧で見えない。
   次の一歩: 親銘柄 ID（V2）の導入。

2. **時間軸が「今」中心**
   全カードが「これから期限切れになるか」を基準に並ぶ。過去の同銘柄との比較や次回予測の動線が無い。
   次の一歩: 銘柄ごとのタイムライン UI（S1）。

3. **「予定」と「実績」が混ざる**
   履歴 = 確定実績のみ。「来週使う予定」を入れる場所が無い。
   次の一歩: UsageEntry に planned / executed フラグ（V4）。

4. **「もらった額面」が記録されない**
   `initial` は「最初の残量」だが、その後 restock すると累計受領額が分からない。
   投資効率（付与 − 消費 − 失効）が計算できない。
   次の一歩: 累計付与額の集計フィールド or 履歴の `kind: "grant"` 追加。

5. **季節 / 期次の trigger が無い**
   ユーザーが「次の権利月だ」と能動的に開く動線をツール側が促せない。
   次の一歩: PWA push（U3）、または起動時バナー（U1）。

## このドキュメントの使い方

- 新機能・データモデル変更・UI 改修の検討時、この一覧で **想定漏れがないか** チェックする
- 実装が進んだら「状態」列を更新する
- 実装方針が固まったものは [decision-log](../decision-log/) に判断理由を残す
- 確定仕様は [Tool Spec](../specs/tools/yutai-memo.md) に反映する

# yutai-expiry 想定優待パターン一覧

## 目的

優待期限帳のデータモデル設計・UI 検討時の前提として、株主優待として実在する代表的なパターンを列挙する。
`unitYen` 必須化（PR #330）など、データモデル制約を入れる際の影響範囲・カバレッジ判断のチェックリストとして使う。

関連:
- [ユースケース棚卸し](./yutai-expiry-usecases.md)
- [UI / data model backlog](./yutai-expiry-ui-data-model.md)

## 前提

- 現在のデータモデルは 2 つの `trackMode` を持つ:
  - **count**: 「N 枚 × 1 枚あたり額面」で価値を表す。`unitYen` 必須。
  - **amount**: 「残高 ¥X」で価値を表す。`unitYen` 不要。
- `count + unitYen=null` は 2026-05-28 PR #330 から validation で禁止された。
- 「額面が確定しない優待」は基本的に **amount モード** で代替する（後述）。

## パターン一覧

### A. 完全カバー: count モード（額面 × 枚数）

各券に明確な額面が印字されており、複数枚束で配布されるタイプ。`count + unitYen` の典型。

| 例 | trackMode | 備考 |
|---|---|---|
| すかいらーく食事優待券（¥500 × N 枚） | count | 半期 / 通期定期 |
| 吉野家・松屋・ゼンショー系食事券（¥500 × N 枚） | count | |
| 居酒屋チェーン食事券（¥500 × N 枚） | count | |
| 物販店割引券（¥1,000 × N 枚） | count | ビックカメラ・ヤマダ電機など |
| 映画館招待券（¥1,800 × N 枚） | count | 額面は実質的に料金相当 |
| 美容院利用券（¥1,000 × N 枚） | count | |

### B. 完全カバー: count モード（額面 × 1 枚）

額面付きの単発券。`count + unitYen` で初期=1 枚。

| 例 | trackMode | 備考 |
|---|---|---|
| QUO カード（¥3,000 / ¥5,000） | count（qty=1） | 単発、額面明確 |
| 図書カード（¥3,000） | count（qty=1） | |
| 商品券（百貨店 ¥1,000） | count（qty=1） | |
| ギフトカード | count（qty=1） | |
| カタログギフト相当額 | count（qty=1） | カタログ全体の額面相当を `unitYen` に入れる |

### C. 完全カバー: amount モード

残高ベースで管理されるタイプ。`amount + balance` で初期残高を入れる。

| 例 | trackMode | 備考 |
|---|---|---|
| WAON / nanaco / 楽天 Edy 等のチャージ式 e-money | amount | restock 操作で残高追加 |
| 電子マネー / ハウスマネー（イオン WAON 株主特典 等） | amount | |
| 自社ポイント残高（無印 MUJI passport 等） | amount | |
| 旅行クーポン残高（HIS, JTB 等） | amount | 利用するたびに金額消費 |
| 株主優待カード割引枠（高島屋 10% off 上限 ¥X） | amount | 上限額を初期残高とみなす |

### D. 一部カバー（amount モードで代替）

額面が明確だが「金額として消費する」用途のもの。count モードでも表現できるが amount の方が自然。

| 例 | 推奨 trackMode | 備考 |
|---|---|---|
| 全国百貨店共通商品券 | amount または count | 額面 ¥1,000 × N 枚なら count、合算管理なら amount |
| プリペイドカード（残高管理型） | amount | |

### E. 想定外 / カバー不能パターン

現在のデータモデルでは扱いが難しい、または非対応のパターン。

| 例 | 課題 |
|---|---|
| **現物送付型優待**（自社製品、米、お菓子等） | 「枚数 × 額面」も「残高」も該当しない。`unitYen` に「目安額」を入れる運用は可能だが正確ではない。**現状は amount モードに目安額を入れて運用** することで救済。または管理しない選択。 |
| **割引券（％ OFF）** | 「20% OFF」は金額換算不能（購入金額に依存）。**割引券は通常本ツール対象外**（=直接の金銭価値が無いため）。記録したい場合は `unitYen=0` で count 管理する救済もあり得るが、validation で 0 が弾かれるため amount + balance=0 のメモ運用を推奨。 |
| **回数券**（10 回入場可など） | 「N 回」は枚数で代用できるが、額面が回数あたりの料金になるため概念的に微妙。`count + unitYen=回数あたり料金` で代用可。 |
| **物理体験権**（ホテル無料宿泊 × 1 泊、施設利用券 × 1 枚） | 額面が「相当額」になる。`count` で扱う、ただし `unitYen` は相場ベース。 |
| **株主限定割引コード**（EC サイトの割引コード） | 1 回のみ使用、額面=割引上限なら count で代用。再使用可なら本ツール対象外。 |
| **配当（金銭）** | 本ツールの対象外（金額の優待ではないため）。別途証券会社のサービスで管理。 |

## 設計判断: unitYen=0 を禁止した理由

PR #330 で `unitYen > 0` を validation 必須にしたため、「額面ゼロの券」は count モードでは扱えない。

**理由**:
- 額面 0 円の券は概念矛盾（無価値の券を集計する意味がない）
- 純粋に「割引券」を管理したい場合は、本ツールの集計から外したい
- 救済が必要なら amount モード + balance=0 + メモ運用で表現可能

## モードの選び方フローチャート

```
優待を登録したい
  ↓
額面が明確に印字されているか？
  ├─ YES
  │   ↓
  │   複数枚配布？
  │     ├─ YES → count モード（qty=N, unitYen=額面）
  │     └─ NO  → count モード（qty=1, unitYen=額面）
  │
  └─ NO（残高型 / チャージ型 / 上限型）
      ↓
      残高金額が分かるか？
        ├─ YES → amount モード（balance=残高）
        └─ NO  → 額面が分かるまで登録を待つ／本ツール対象外
```

## バリデーション制約まとめ（2026-05-28 時点）

| 項目 | 制約 | 例外 |
|---|---|---|
| count: qty | 0 以上 | qty=0 は pending 登録として許容 |
| count: unitYen | qty > 0 のとき必須、0 より大きい数値 | qty=0 なら省略可 |
| amount: balance | 0 以上の数値 | 0 もメモ用途として許容 |
| title / company | 必須 | — |
| expiresOn | 任意（YYYY-MM-DD 形式） | 期限なし優待のため空可 |

## 今後の検討

- E のパターン（割引券・回数券・現物送付）を本ツールで管理するべきか、別ツールに切り出すか
- 「目安額」を扱うフラグ（`unitYen` に近似値を入れたことを明示するため）
- カタログギフトの「複数選択肢からどれを取った/取らなかった」の表現

これらは [ユースケース棚卸し](./yutai-expiry-usecases.md) の O. 拡張機能候補 として扱う。

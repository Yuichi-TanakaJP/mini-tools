# 2026-05-04 yutai-memo と優待カレンダー連携フィールド整理

## 背景

`yutai-candidates`（優待カレンダー）から `yutai-memo`（優待銘柄メモ帳）へ銘柄を追加できる。
一方で、同じ銘柄を手動で先にメモしている場合に、カレンダー追加や将来のマージ機能で手動メモが上書きされないか確認する必要が出た。

このメモでは、2026-05-04 時点の実装事実と、将来「既存メモにカレンダー情報をマージする」場合のフィールド別ポリシーを整理する。

## 結論

- 現状は、同じ `code + month` のメモが既にある場合、追加処理は止まり、既存メモは変更しない。
- 現状は、同じ `code` でも `month` が違う場合、別メモとして新規追加される。
- 将来マージ機能を入れる場合、`memo` 本文やタグなどの手動管理フィールドは上書きしない。
- カレンダー由来のメタ情報は、手動メモと分離して保存・表示する。
- `relatedUrl` はメモ側の公式/主要リンクとして扱うが、ユーザーが編集できるフィールドなので、既存値がある場合は自動上書きしない。

## 現状の実装

### 追加元

- `app/tools/yutai-candidates/ToolClient.tsx`
  - `handleAdd()` で優待カレンダーの銘柄をメモへ追加する。
- `app/tools/yutai-memo/candidate-import.ts`
  - `isImportedMonthlyYutaiCandidate()` で重複判定する。
  - `createMemoItemFromCandidate()` でカレンダー由来の `MemoItem` を作る。
  - `addMemoItemFromCandidate()` で LocalStorage へ保存する。
- `app/tools/yutai-memo/storage.ts`
  - `yutai_memo_items_v1` に保存する。

### 重複判定

現状の重複判定は次の条件。

```ts
item.code === candidate.code && item.months.includes(candidate.month)
```

つまり、同じ銘柄コードかつ同じ権利月が既に含まれているメモは「追加済み」とみなす。

### 現状の保存動作

| 状況 | 現状の動作 | 既存メモへの影響 |
|---|---|---|
| 同じ `code + month` のメモがない | 新しい `MemoItem` を先頭に追加 | 既存メモは変更しない |
| 同じ `code + month` のメモがある | 追加せず「すでに優待メモにあります」扱い | 既存メモは変更しない |
| 同じ `code` だが別 `month` のメモがある | 別メモとして新規追加 | 既存メモは変更しない |
| 手動メモに `code` がない | 重複判定できないため新規追加 | 既存メモは変更しない |

## フィールド一覧と扱い

### 手動管理フィールド

これらはユーザーがメモ画面で編集する領域。将来のカレンダー情報マージでも、自動上書きしない。

| フィールド | 用途 | 現状の画面表示 | カレンダー新規追加時 | 将来マージ時の推奨 |
|---|---|---:|---|---|
| `name` | 銘柄名 | 表示/編集あり | カレンダーの会社名を設定 | 既存値を維持。空なら補完可 |
| `code` | 銘柄コード | 表示/編集あり | カレンダーのコードを設定 | 既存値を維持。空なら補完可 |
| `months` | 権利月 | 表示/編集あり | `[candidate.month]` を設定 | 同じメモへ統合する設計にする場合だけ追記。自動統合は慎重に扱う |
| `tagIds` | ユーザータグ | 表示/編集あり | 空配列 | 変更しない |
| `crossType` | 戦略タイプ | 表示/編集あり | `長期優遇なし` | 変更しない |
| `entryTiming` | 早打ち目安 | 表示/編集あり | 未設定 | 変更しない |
| `tenureRule` | 任期条件 | 表示/編集あり | 未設定 | 変更しない |
| `acquired` | 取得済み状態 | 表示/編集あり | `false` | 変更しない |
| `oneShareStartedAt` | 1株保有開始時期 | 表示/編集あり | 未設定 | 変更しない |
| `priority` | 優先度 | 一部UIで利用 | `2` | 変更しない |
| `memo` | 自由メモ本文 | 表示/編集あり | カレンダー由来の初期メモを生成 | 既存本文は上書きしない。自動追記も原則しない |
| `relatedUrl` | メモ側の公式/主要リンク | 表示/編集あり | `officialBenefitUrl` 優先、なければ `minkabuYutaiUrl` | 既存値が空のときだけ補完。既存値は自動上書きしない |

### カレンダー由来メタフィールド

これらは優待カレンダーから来る情報。現状は保存されるが、メモ画面ではほぼ表示されていない。
将来は「カレンダー情報」ブロックとして表示すると、手動メモ本文と混ざらず比較しやすい。

| フィールド | 用途 | 現状の画面表示 | カレンダー新規追加時 | 将来マージ時の推奨 |
|---|---|---:|---|---|
| `source` | 追加元データソース | 表示なし | `minkabu` | カレンダー由来値として更新可 |
| `pickedFrom` | 追加元画面 | 表示なし | `monthly_yutai_list` | カレンダー由来値として更新可 |
| `minimumInvestmentText` | 最低投資金額テキスト | 表示なし | カレンダー値を保存 | カレンダー由来値として更新可 |
| `benefitCategoryTags` | 優待カテゴリ | 表示なし | カレンダー値を保存 | カレンダー由来値として更新可 |
| `minkabuYutaiUrl` | みんかぶ優待ページ | 表示なし | カレンダー値を保存 | カレンダー由来値として更新可 |
| `officialBenefitUrl` | 公式優待ページ | 表示なし | カレンダー値を保存 | カレンダー由来値として更新可 |
| `officialLinkStatus` | 公式リンク確認状態 | 表示なし | カレンダー値を保存 | カレンダー由来値として更新可 |

## 上書き・追記ポリシー案

将来「既にあるメモへカレンダー情報を反映する」ボタンを作る場合は、次のルールを基本にする。

| 種別 | 対象 | ポリシー |
|---|---|---|
| 手動本文 | `memo` | 上書きしない。自動追記もしない |
| 手動分類 | `tagIds`, `crossType`, `priority`, `acquired`, `oneShareStartedAt` | 変更しない |
| 手動補足 | `entryTiming`, `tenureRule` | 変更しない |
| 主要リンク | `relatedUrl` | 空なら補完。値があれば維持 |
| カレンダー情報 | `minimumInvestmentText`, `benefitCategoryTags`, `minkabuYutaiUrl`, `officialBenefitUrl`, `officialLinkStatus`, `source`, `pickedFrom` | カレンダー由来として更新可 |
| 銘柄基本情報 | `name`, `code` | 空なら補完。既存値があれば維持 |
| 権利月 | `months` | 同一 `code` の単一メモへ統合する仕様を採る場合のみ追記。現状の別メモ運用を変えるなら別途判断する |

## UI 方針案

### 0. 2026-06-20 追加済みメモの簡易編集

優待カレンダー上で同じ `code + month` のメモがある場合、従来の `追加済み` 固定表示ではなく `メモ編集` ボタンを表示する。
クリックすると、該当 `MemoItem` を LocalStorage から読み直し、次の手動管理フィールドをその場で編集・保存できる。

- `name`
- `crossType`
- `entryTiming`
- `relatedUrl`
- `tenureRule`
- `acquired`
- `priority`
- `memo`

`months` はカレンダー側では変更しない。
理由は、カレンダーの追加済み判定が `code + month` であり、この画面で権利月を外すとカード状態と保存データの対応が分かりにくくなるため。
権利月・タグ・1株保有開始時期などの詳細編集は、引き続き優待メモ帳本体で扱う。

誤って追加した場合は、同じ編集ダイアログから `追加を解除` できる。
解除時は確認を挟み、対象 `MemoItem` が単月ならメモ自体を削除し、複数月を持つ場合は表示中の `month` だけを `months` から外す。
これにより、カレンダーの `code + month` 追加済み状態を未追加に戻しつつ、他の権利月メモは残せる。

2026-06-20 追記: 優待メモへ追加する前の判断材料として、優待カレンダーのカード自体にカレンダー内メモを持たせる。
保存先は `monthly_yutai_card_memos_v1` とし、`code + month` ごとに `longTermRequired` / `longTermBenefit` を保存する。
これは優待メモ帳の `MemoItem` とは別管理にする。
理由は、まだ優待メモへ追加するか決めていない候補にも「長期保有が要」「長期優待あり」などを残したいため。
カードの縦幅を増やさないため、専用メモ欄は持たせず、銘柄名行の小さなチップでトグルする。
長期フラグは検索対象にも含める。

### 1. ボタン表示

優待カレンダー側では、同じ `code + month` のメモがある場合に次のどちらかを表示する。

- 現状維持: `追加済み`
- 将来案: `カレンダー情報を更新`

`カレンダー情報を更新` は、手動フィールドを変更しないことが分かる文言にする。

### 2. メモ画面の表示

カレンダー由来メタ情報を表示する場合は、自由メモ本文とは分ける。

表示例:

- 最低投資金額: `minimumInvestmentText`
- 優待カテゴリ: `benefitCategoryTags`
- 公式優待ページ: `officialBenefitUrl`
- 参考リンク: `minkabuYutaiUrl`
- 公式リンク状態: `officialLinkStatus`

### 3. 編集保存時の注意

2026-05-04 時点の `yutai-memo` 編集保存処理は、編集フォームに存在するフィールドを元に `MemoItem` を作り直している。
そのため、画面に出ていないカレンダー由来メタフィールドは、編集保存時に落ちる可能性がある。

カレンダー由来メタフィールドを正式仕様として使う場合は、編集保存時に既存のメタフィールドを保持する実装へ変更する。

## 影響範囲

- `/tools/yutai-candidates`
  - メモ追加ボタンの重複判定と将来の更新ボタン表示。
- `/tools/yutai-memo`
  - カレンダー由来メタ情報の表示。
  - 編集保存時のメタフィールド保持。
- LocalStorage
  - `yutai_memo_items_v1`
  - 既存データとの互換性を維持する必要がある。

## 残課題

- 同じ `code` で別 `month` のメモを、将来ひとつのメモへ統合するかどうか。
- カレンダー由来メタ情報をカード一覧にどこまで表示するか。
- `officialBenefitUrl` と `relatedUrl` の表示上の役割分担。
- 既存メモにカレンダー情報を更新した場合の通知文言。

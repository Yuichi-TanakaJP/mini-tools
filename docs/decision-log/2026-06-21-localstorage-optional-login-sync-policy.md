# 2026-06-21 ローカルデータの任意ログイン同期方針（オプトイン・クロスデバイス）

## 背景

- mini-tools の LocalStorage 系ツール（優待メモ帳・優待カレンダー・優待期限帳・マイ銘柄など）は、ログイン不要・端末完結で使えることを長所にしてきた。
- 一方で「機種変更・ブラウザ変更・サイトデータ削除でデータが消える」「同じ人が複数端末で同じ状態を見られない」という弱点がある。
- ユーザー要望として「ログインしなくても今まで通り使えるが、希望者はログインしてデータを残せる／端末をまたいで使える、を選べるようにしたい」という方向が出た。
- 既存の認証は `premium` 配下の共有パスワード方式（[2026-04-04 premium ログイン導線の暫定実装方針](./2026-04-04-premium-login-placeholder-flow.md)）であり、ユーザー単位の識別はまだない。

このメモは、2026-06-21 時点で固めた「任意ログインによるクロスデバイス同期」の設計方針を残す。実装はまだ着手していない。

## 今回決めたこと

### 方針（結論）

- **オプトイン同期**にする。未ログインのままでも従来通り LocalStorage で完結して使える。
- ログインは「データを端末の外に残す／端末をまたぐ」ための任意機能と位置づける。ログインしただけでデフォルト挙動を変えない。
- ログインしても **LocalStorage は捨てない**。LocalStorage を手元のキャッシュ兼オフライン動作の正として使い続け、サーバーとは裏で同期する（ローカルファースト）。
- 同期対象は優待ツールに限定せず、**LocalStorage 系ツール全般**に広げられる汎用設計を前提にする。各ツールの保存キー（例: `yutai_memo_items_v1`, `monthly_yutai_card_memos_v1`, `monthly_yutai_picks_v1` / `monthly_yutai_passes_v1` など）を同期ユニットとして登録できる形にする。

### 段階移行

big-bang で全部作らず、次の段階で進める。

- **Phase 0（救済を最短で）**: ログインなしのまま、各ツールに JSON エクスポート / インポートを用意する。機種変更時の手動移行を先に成立させる。同期はしない。
- **Phase 1（任意ログイン + サーバー保存）**: ユーザー単位の認証を入れ、ログイン時に同期対象キーをサーバーへ保存・復元できるようにする。まずは手動同期（保存/復元ボタン）でよい。
- **Phase 2（自動同期の洗練）**: 自動同期と競合解決を整える。

### 設計の前提ルール

- **移行（初回ログイン時の吸い上げ）**: 既にローカルにデータがある人が初めてログインしたとき、そのローカルデータをサーバーへ取り込む導線を用意する。空のサーバーで上書き消去しない。
- **競合解決**: 既存データの多くが `updatedAt` を持つため、当面は last-write-wins（新しい方優先）を基本とする。要素単位の `updatedAt` がないキーは、Phase 1 ではキー丸ごとの新旧比較で割り切る。
- **ログアウト時の扱い**: ローカルデータを残すか消すかを選べるようにする。共用端末配慮として「ログアウト時にこの端末のローカルデータを消す」選択肢を持たせる。
- **プライバシー**: 現状は端末内完結で何も預けていない。サーバー保存は「個人の銘柄ポートフォリオ等を預ける」ことになるため、何を保存するか・任意であることを明示する。

## 判断理由

- 「ログイン不要で軽い」という現状の長所を失わずに、消失リスクと端末間共有という弱点だけを補える。モダンなローカルファースト構成の定石でもある。
- LocalStorage を残してキャッシュ扱いにすることで、表示速度・オフライン耐性・既存実装（SSR/hydration 前提、[2026-03-12 SSR / Hydration / localStorage 運用ガイド](./2026-03-12-ssr-localstorage-hydration-guidelines.md)）を壊さずに済む。
- 認証 + DB + 同期は大きい機能なので、Phase 0 の JSON エクスポート/インポートを先に出せば、最小コストで「機種変更で消える」問題に手を打てる。
- 競合解決を最初から完璧にすると重い。`updatedAt` ベースの last-write-wins で割り切り、必要になってから精緻化する。

### 採らなかった案

- **ログイン必須に切り替える**: 現状の「ログイン不要」という価値を失うため不採用。
- **ログインしたら LocalStorage をやめてサーバー専用にする**: オフライン動作と表示速度を損なうため不採用。キャッシュとして併用する。
- **既存の premium 共有パスワード方式を流用する**: ユーザー単位の識別ができず、個人データの持ち主を区別できないため、同期用途には使えない。ユーザー単位認証の導入が別途必要。

## 影響範囲

- 現状の Tool Spec は「サーバー DB に保存しない／端末ローカルで完結」を明記している（[優待銘柄メモ帳 仕様](../specs/tools/yutai-memo.md) の保存先節など）。Phase 1 着手時は、この記述を「未ログイン時はローカル完結／ログイン時は任意でサーバー同期」に更新する必要がある。
- 認証基盤: ユーザー単位の認証（サインアップ/ログイン）が新規に必要。既存 premium 認証とは別レイヤー。
- データ層: 同期対象キーごとのサーバー保存先（DB）と同期 API。
- 各ツール Client: 同期対象キーの登録、エクスポート/インポート UI、ログイン状態に応じた保存先切替。
- 互換性: 既存 LocalStorage データ形式を壊さない。サーバーは既存キーの値をそのまま預かる形から始める。

## 残課題

- ユーザー単位認証の方式（自前 / 外部 IdP）と、premium 認証との統合か併存かの整理。
- サーバー保存先（DB 種別・スキーマ）と同期 API の設計。
- 競合解決を要素単位に精緻化するか、キー丸ごと last-write-wins で続けるかの線引き。
- 同期対象キーの正式な一覧化（どのツールのどのキーを対象にするか）。
- プライバシー表記・利用規約・データ削除（退会時の消去）導線。
- Phase 0 の JSON エクスポート/インポートを共通コンポーネント化するか、ツール個別に持つか。

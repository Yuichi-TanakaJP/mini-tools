# 2026-06-22 ログイン直後の自動同期を止める判断

## 背景

- 2026-06-21 に Supabase Auth + `tool_data` による任意ログイン同期を本番へ入れた。
- その後、同期対象キーを広げた際に、空配列・空オブジェクトを正しい最新データとして扱う経路があり、端末内データやサーバーデータを空で上書きできてしまった。
- バグ自体は空データ上書き防止と安全バックアップで修正したが、ログイン直後に `pull -> push` を自動実行する仕様は、ユーザーが保存/復元の意図を選ぶ前にデータを動かすため、同期基盤が成熟するまでリスクが高い。
- 他アプリの一般的な方針も、同期対象の選択、明示的な保存/復元、競合時の保全を重視している。

## 今回決めたこと

- `/account` のログイン直後・新規登録直後には、クラウド保存/復元を自動実行しない。
- ログインは認証状態を作るだけにし、同期はユーザーがアカウント画面で明示的に選ぶ。
- Phase 1 の同期 UI は次の手動操作を基本にする。
  - `この端末を保存`: 端末内の実データをクラウドへアップロードする。
  - `クラウドから復元`: クラウド側のデータをこの端末へ反映する。
- 空データによる上書き防止と安全バックアップは維持する。

## 判断理由

- mini-tools はログイン不要・端末内保存で使えることが価値なので、ログインしただけでデータ移動が起きると期待とずれやすい。
- 競合解決 UI、差分比較、クラウド側の履歴復元がまだない段階では、自動同期より明示操作の方が事故時の原因を追いやすい。
- Phase 1 の当初計画にも「まずは手動同期でよい」「Phase 2 で自動同期」としており、今回の変更はその段階設計に戻すもの。
- 自動同期を将来再導入する場合も、端末ごとの最終同期状態、競合表示、復元履歴、ユーザー設定を揃えてからにする。

## 影響範囲

- `app/account/AccountClient.tsx`
  - `syncOnLogin()` を廃止。
  - ログイン/新規登録後のメッセージを、手動保存/復元の案内に変更。
  - 同期カードの説明を「ログイン直後に自動同期しない」に変更。
- `lib/sync/client.ts`
  - 既存の `pushAll` / `pullAll` は維持。手動ボタンからのみ使う。
- `docs/plans/phase1-cross-device-sync-plan.md`
  - クライアント同期層の Phase 1 記述を手動同期中心へ更新。

## 残課題

- 「同期状況を確認」表示（最終保存時刻、クラウド側件数、端末側件数）は未実装。
- 自動同期を再導入する条件は未実装。Phase 2 で競合解決 UI と復元履歴を含めて再検討する。
- 既に消えた端末データの復旧は、この仕様変更だけではできない。Supabase 側またはブラウザ内安全バックアップに残っている場合だけ復旧余地がある。
